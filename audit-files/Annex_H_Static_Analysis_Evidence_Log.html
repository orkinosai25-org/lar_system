4. CSRF Protection Disabled — All Modules P1

Command: grep -rn "csrf_protection" --include="config.php"

# All six modules: services/webservices/application/config/config.php:297 $config['csrf_protection'] = FALSE; agent/application/config/config.php:297 $config['csrf_protection'] = FALSE; supplier/application/config/config.php:297 $config['csrf_protection'] = FALSE; b2c/config/config.php:297 $config['csrf_protection'] = FALSE; ultralux/application/config/config.php:297 $config['csrf_protection'] = FALSE; supervision/application/config/config.php:297 $config['csrf_protection'] = FALSE;

CSRF protection is disabled on all CodeIgniter configurations. POST-based booking, payment, and management forms are vulnerable to Cross-Site Request Forgery attacks. An attacker on a third-party site can trigger booking or profile changes on behalf of a logged-in user.

Remediation: Set $config['csrf_protection'] = TRUE; across all modules and include {csrf_field()} in all POST forms.

5. Cookie Security Flags Disabled P1

Command: grep -rn "cookie_secure\|cookie_httponly" --include="config.php"

# Across all six modules: */config/config.php:272 $config['cookie_secure'] = FALSE; # HttpOnly flag — not explicitly set in any module config (defaults may vary)

cookie_secure = FALSE means session cookies are transmitted over plain HTTP, exposing them to network eavesdropping (session hijacking). The Secure, HttpOnly, and SameSite cookie attributes must be enforced on all session cookies.

Remediation: Set $config['cookie_secure'] = TRUE; and verify cookie_httponly = TRUE in all modules. Deploy behind HTTPS-only (enforce HSTS).

Verbose PHP errors in customer-facing booking, payment, and flight controllers expose internal stack traces, file paths, SQL queries, and application structure to end users and attackers. This is a P0 risk (Audit v7.3.3 §1 — Verbose error exposure in payment/booking flows).

Remediation: Set ini_set('display_errors', 0); globally in production index files. Route all errors to a secure log file only. Remove all per-controller error_reporting(E_ALL) / display_errors=1 calls.

7. Weak Password Hashing (MD5) P0

Command: grep -rn "md5(" --include="*.php" | grep -i password

# Authentication using MD5 — cryptographically broken (rainbow-table / collision attacks) services/webservices/application/hooks/hooks.php:115 'password' => md5($password) services/webservices/application/hooks/hooks.php:167 'password' => md5($this->CI->db->escape_str($password)) services/webservices/application/models/user_model.php:40 'password' => md5($password) agent/application/controllers/auth.php:66 $user_record['data'][0]['password'] = provab_encrypt(md5('Provab@123')); supervision/application/controllers/general.php:225 $update['password'] = provab_encrypt(md5('Provab@123'));

MD5 is not a password hashing algorithm. It is a fast hash designed for checksums, making brute-force and rainbow-table attacks trivially fast. All user authentication must migrate to password_hash() with PASSWORD_ARGON2ID or PASSWORD_BCRYPT with progressive rehashing on next login.

"',512)) AS decrypt_data") b2c/controllers/general.php:492, 521 // Same AES_DECRYPT pattern repeated — $encrypt_key interpolated directly into query string agent/application/hooks/hooks.php:327 $this->CI->db->query("SELECT AES_DECRYPT($encrypt_key,SHA2('" . $md5_key . Multiple locations pass user-controlled or session-derived values directly into SQL query strings without parameterisation. The CodeIgniter Query Builder and prepared statements ($this->db->query($sql, $params)) should be used consistently. The AES_DECRYPT pattern is particularly sensitive as it occurs in authentication/decryption flows.

9. XSS Risk — Unescaped $_GET Output P1

Command: grep -rn "echo.*\$_GET" --include="*.php"

# $GET values echoed without htmlspecialchars / form_input escaping: supplier/application/views/.../report/b2c_hotel_report.php:16 echo $_GET['app_reference']; // in form input value attribute supplier/application/views/.../report/b2c_hotel_report.php:31 echo $_GET['created_datetime_from']; supplier/application/views/.../report/b2c_hotel_report.php:32 echo $_GET['created_datetime_to']; supervision/.../utilities/manage_promo_code.php:64 echo generate_options($options, (array)@$_GET['module']);

User-supplied GET parameters are echoed directly into HTML attributes in report search forms and the promo-code admin page without escaping. An attacker can inject arbitrary JavaScript via a crafted URL. All output of untrusted data must be wrapped with htmlspecialchars() or CodeIgniter's html_escape().
