5. Residual Risks to Acceptance

The following risks, in priority order, could lead to a second rejection. Each has a recommended mitigation.

Risk R-1 (High) — Static-Only Evidence vs. Pre-Production Runtime Requirement

Description: LAR's cure condition (a) and TOR §5.5 explicitly require evidence from "the agreed pre-production environment." All evidence in the current submission derives from static source code analysis. No pre-production environment, live server logs, GDS sandbox, or API trace capture was made available during the engagement.

Impact if unmitigated: LAR could reject the submission on the same ground as the original non-acceptance notice (1.1 — "no evidence for P0/P1 findings"), arguing that the static analysis evidence does not satisfy the contractual definition.

Mitigation: The constraint is documented in Annex I (C-001–C-005) and referenced in every finding. In the submission cover communication, explicitly note: "The auditor has produced the maximum evidence achievable within the access constraints of this engagement. Static analysis is the highest-fidelity evidence available when no pre-production environment access is provided. Annex I documents all constraints. Runtime confirmation of each finding is pending LAR's provision of the pre-production environment and GDS sandbox credentials." The 5-day alignment call option offered by LAR should be used to resolve this position before final submission.

Risk R-2 (Medium) — "Test Scenario Outcomes" for Verticals

Description: Cure condition (b) asked for "actual test scenario outcomes" for Flights, Hotels, and Cars. The v7.4 vertical findings (Sections 3–5) are code-derived observations (static), not outcomes of executed booking scenarios.

Mitigation: Annex I documents that no GDS or supplier API sandbox was provided (C-002, C-003). The code-level findings are real and evidenced (file path + line reference). The Annex I per-vertical test matrices (T-001–T-014) show precisely which scenarios remain pending environment access. Communicate that the findings represent code evidence of functional gaps — the missing "test outcome" is that the test cannot be run because the implementation is absent.

Risk R-3 (Low) — Annex F Wording Inconsistency

Description: AUDIT_REPORT.md §Annexes states "The Visual Audit Guide (previously submitted as Annex F) has been removed from this revised submission." However, the revised file audit-files/Annex F — Visual Audit Guide.html still exists in the repository (though all commercial pricing has been removed). If LAR reads the report text and then sees the file still present, this inconsistency could create confusion or the appearance of incomplete action.

Mitigation (applied in this PR): Update the Annex F note in AUDIT_REPORT.md to clarify that the file has been revised (not deleted) to remove all commercial content per SOW §6a, and is retained as an optional supplemental visual reference. See Section 7 below for the specific change.

Risk R-4 (Low) — Revenue Figures May Still Draw Scrutiny

Description: The replacement revenue analysis (Section 7) uses IATA and Phocuswright 2024 benchmark references. LAR's CTO may ask for the specific citations.

Mitigation: Section 7 is clearly labelled "Low" confidence with an explicit statement that all figures must be validated against LAR's actual transaction data. The methodology is transparent. If LAR requests specific citations, provide the IATA 2024 Airline IT Insights report and Phocuswright "Online Travel in Focus" 2024 as the benchmark sources.

6. Recommendations Before Submission

6.1 Immediate Actions (Before Sending)

Action 1 — Use the alignment call option.
LAR's notice offered a brief alignment call within 2 business days "if you have questions about any of the specific requirements listed above." Given that Risk R-1 (environment access) is a structural constraint — not a content gap — this call should be requested immediately. The objective is to either: (a) secure environment access before the cure deadline, or (b) obtain LAR's written acknowledgement that static-analysis evidence is acceptable given the constraint. This protects both parties.
Action 2 — Write a concise, structured cover letter.
Do not submit the documents without a cover letter that:
  • Confirms receipt of and response to the non-acceptance notice
  • Maps each of the 9 cure conditions to its location in the revised submission
  • Clearly states the environment access constraint (Annex I C-001) and explains that static analysis is the maximum evidence achievable without that access
  • Requests LAR to provide pre-production environment and GDS sandbox access to allow the 14 pending test scenarios (Annex I, T-001–T-014) to be executed for a final evidence-complete submission
Action 3 — Fix the Annex F description inconsistency in AUDIT_REPORT.md.
Update the Annex F note to say the file has been revised (commercial content removed) rather than removed entirely. This avoids confusion and is more accurate. This change is applied in this PR.

6.2 What NOT to Do

  • Do not reintroduce commercial pricing, ROI projections, or remediation bundles anywhere in the submission. This would re-trigger SOW §6a violation.
  • Do not reintroduce out-of-scope content (UI/UX redesign, AI Chat, Cruises, Aviation, Yachts).
  • Do not reinstate the unsupported $6.8M revenue figure or any revenue estimate without the explicit methodology now present in Section 7.
  • Do not submit without the cover letter. Bare document submission leaves the residual risks unaddressed.
